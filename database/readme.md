# Проектирование БД

## Концептуальная модель

Концептуальная модель - это схема структуры базы данных, спроектированная без привязки к какой-либо СУБД, целью которой является донести до всех, кто будет работать с базой данных её цель и вид, а также определить ключевые сущности их атрибуты и связи

Система должна обеспечить сотрудникам возможность быстрой адаптации в коллектив, а руководителям экономить время и деньги на обучение новичков. 

При первом входе в систему работнику потребуется пройти промежуточное тестирование для выявления дыр в знаниях, и аналитики, что нужно будет пройти, и почему. 

### Ключевые отношения

На этапе проектирования я думаю стоит рассмотреть базу данных как единую, без разделения на отдельные базы и схемы. Для галочки я буду стараться делить по доменам сущности, которые буду описывать ниже.

#### iam_schema

Identity and access management schema. Не скажу, правильно ли так связывать auth и user, но это мое приложение, и в рамках него я думаю, что такое решение скажется только положительно.

Отношения: 

1. `users` - отношение, отвечающее за авторизационные вопросы.
2. `roles` - отношение, хранящее роли сотрудников.
3. `permissions` - отношение, в котором хранятся все разрешения
4. `roles_permissions` - отношение, разрешающее связь `n * n` между `roles` и `permissions`
5. `user_permissions` - отношение, в котором хранятся специфичные для пользователя разрешения

#### company_schema

Схема для структуры компании. Employee существует лишь в рамках компании, поэтому он тоже лежит тут. Я не люблю до крошек все разделять.

Отношения: 

1. `employess` - отношение, в котором хранятся данные о сотруднике, по типу employment_date.
2. `departments` - отношение, в котором хранятся отделы компании.
3. `position` - отношение, в котором хранятся конкретные должности компании.

#### learn_schema

- Курсы 

Курс является ключевой сущностью системы, вокруг которой сосредоточится и вся остальная логика. Из чего состоит курс:

- Базовые атрибуты, по типу имени, описания
- Модули, на которые разбивается курс. Например, курс JS, модуль 1 - введение в программирование. Состоит из имени
- Тесты. Тест - особая состоящая курса. Состоит из самого теста, вопросов к тесту, вариантов ответа. Связи один тест - много вопросов, один вопрос - много ответов. Для ответов сотрудников будет также сущность. Тест связан с модулем, и также будет иметь атрибут тип, чтобы можно было понимать, default, между лекциями курса, between - между модулями, и какой-нибудь final, после прохождения курса. 

Отношения

1. `courses` - отношение, в котором хранятся курсы.
2. `course_requests` - отношение, хранящее запросы на доступ к курсу. То есть сотрудник одного отдела может захотеть переквалифицироваться, и запросить доступ к курсу другого отдела.
3. `course_tests` - отношение для тестов, которые относятся к модулю курса. Имеет тип.
4. `course_questions` - вопросы к тесту. многие к 1 с `course_tests`
5. `course_answers` - варианты ответов к тесту. многие к 1 с `course_questions`.
6. `test_results` - результаты прохождения тестов. Данные только создаются, update нельзя для статистики.  

`tasks` - отношение, в котором руководитель может назначать курсы для прохождения и планировать созвоны.
